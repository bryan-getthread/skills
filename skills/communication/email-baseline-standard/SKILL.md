# Email Baseline Standard

The shared foundation for all client-facing email skills. Other skills (client-reply, outage-notification, resolution-closing-email, and the rest of the communication category) reference this standard rather than restating it. When any skill says "per the Email Baseline Standard," apply everything below.

## When to use

- Automatically, whenever another communication skill references it.
- Directly, when asked "what's our email standard?" or when drafting an email no specialized skill covers.

## The standard

### Structure (in order)

1. **Greeting** — client's first name, first message of a thread only. No greeting on subsequent replies.
2. **Context line** — one sentence anchoring which issue/ticket this is about, so the email stands alone in their inbox.
3. **Body** — the substance, answer first, then detail. 1–2 short paragraphs for routine messages.
4. **Next steps** — who does what next, and by when. If the client must act, make the action unmissable (its own line or a short list).
5. **Close** — a plain forward-looking sentence. No sign-off block if a managed signature is applied automatically.

### Tone rules

- Plain, direct, warm. Write like a competent human, not a template.
- No filler openers or hedging padding ("I hope this finds you well", "Just circling back").
- No jargon unless the client used the term first; gloss unavoidable technical terms in plain language.
- Never blame the client, a colleague, a vendor, or a previous tech.
- Commit only to what is confirmed. "We'll update you by <time>" beats "this should be fixed soon."

### Placeholder discipline

- Drafts may use bracketed placeholders — <client first name>, <ticket number>, <time window> — but **every bracket must be filled before sending**. A skill producing a draft must either fill each placeholder from ticket data or flag the unfilled ones explicitly at the top of its output ("NEEDS: <time window>").
- Never let a placeholder leak into a sent message.

### Defensive writing

- State facts, not speculation. Never assert a cause, a breach, or a third party's failure that isn't confirmed in the ticket record.
- Never use "hacked," "breached," or "compromised" for unconfirmed security events — say "suspicious activity we are investigating."
- Write every email as if it will be forwarded to the client's lawyer, their boss, and your competitor — because sometimes it is.

## Guardrails

- This skill defines standards; it does not send anything.
- House-specific overrides (per-partner tone, banned words, signature policy) take precedence over this baseline — apply them on top.
- Localizable: these rules survive translation; when drafting in another language, keep structure and defensive-writing rules intact.
